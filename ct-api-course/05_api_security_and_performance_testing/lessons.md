# Chapter 5 — API Security and Performance Testing: Lessons

## 5.1 API Authentication Mechanisms

### Basic Authentication

**How it works:**
- Username and password combined with a colon (`user:pass`)
- Base64 encoded
- Sent in the `Authorization` header: `Basic dXNlcjpwYXNz`

**Characteristics:**
- Simple to implement
- Credentials sent with every request
- Must use HTTPS (plaintext in HTTP)
- No session management

**Testing considerations:**
- Verify HTTPS is enforced
- Test with missing/invalid credentials
- Verify 401 for missing auth
- Test for brute force protection

---

### Bearer Token / API Key

**How it works:**
- Token or key generated by the API provider
- Sent in the `Authorization: Bearer <token>` or custom header (e.g., `X-API-Key`)
- API validates token/key on every request

**Characteristics:**
- Simple for clients
- Token can be revoked
- Rate limiting typically applies per key
- No built-in expiration (unless implemented)

**Testing considerations:**
- Test with missing, invalid, and expired tokens
- Verify rate limiting per key
- Test token revocation
- Verify different keys have appropriate permissions

---

### OAuth2

**How it works:**
1. Client requests authorization from resource owner (user)
2. Client exchanges authorization grant for access token
3. Client uses access token to access protected resources

**Flows (Grant Types):**

**Authorization Code Flow (most secure):**
- Best for server-side applications
- Redirect user to authorization server
- Exchange code for token
- Refresh tokens supported

**Client Credentials Flow:**
- For machine-to-machine communication
- Client authenticates directly
- No user involved

**Password Grant (legacy):**
- Client collects username/password
- Discouraged for new applications

**Testing considerations:**
- Verify token expiration and refresh
- Test with expired tokens
- Verify scope enforcement
- Test CSRF protection in authorization code flow
- Verify PKCE for public clients

---

### JWT (JSON Web Token)

**Structure:**
```
header.payload.signature
```

**Header:** Algorithm and token type
**Payload:** Claims (user ID, expiration, scopes)
**Signature:** HMAC or RSA signature for verification

**Characteristics:**
- Self-contained (no server-side session needed)
- Can be verified without contacting authorization server
- Has expiration built-in (`exp` claim)
- Tamper-evident (signature)

**Testing considerations:**
- Verify signature validation (tampered token should fail)
- Test expired JWTs
- Verify `iss`, `aud`, `sub` claims
- Test with missing/ malformed JWTs
- Verify algorithm is not `none`

## 5.2 Common API Security Vulnerabilities

### OWASP API Security Top 10

**API1: Broken Object Level Authorization**
- User can access other users' data by changing IDs in requests
- **Example:** `GET /users/123/orders` → change to `124` and access another user's orders
- **Test:** Send requests with different IDs and verify access control

**API2: Broken Authentication**
- Weak or missing authentication
- Predictable tokens, weak passwords allowed
- No brute force protection
- **Test:** Attempt authentication attacks (brute force, token guessing)

**API3: Excessive Data Exposure**
- API returns more data than needed
- Client filters data, but API exposes everything
- **Example:** User profile API returns password hash, internal IDs, SSN
- **Test:** Inspect full response for sensitive fields

**API4: Lack of Resources and Rate Limiting**
- No limits on request frequency or payload size
- Enables DoS attacks
- **Test:** Send requests rapidly, send oversized payloads

**API5: Broken Function Level Authorization**
- Regular users can access admin endpoints
- Missing role-based access control
- **Test:** Access admin endpoints with regular user credentials

**API6: Mass Assignment**
- Client can set fields that should not be user-modifiable
- **Example:** Setting `isAdmin: true` in a user registration request
- **Test:** Send requests with extra fields not in the UI

**API7: Security Misconfiguration**
- Default credentials, unnecessary features enabled
- Missing security headers, verbose error messages
- **Test:** Check security headers, verify no default credentials

**API8: Injection**
- SQL injection, NoSQL injection, command injection, LDAP injection
- **Example:** `{"username": "admin' OR '1'='1"}`
- **Test:** Send injection payloads in all input fields

**API9: Improper Assets Management**
- Old API versions still accessible with known vulnerabilities
- Debug endpoints left in production
- **Test:** Check for old API versions, debug endpoints

**API10: Insufficient Logging and Monitoring**
- No audit trail of API access
- Slow detection of attacks
- **Test:** Verify access logs exist and contain relevant information

## 5.3 API Security Testing Techniques

### Authentication Testing

**Test cases:**
1. Missing authentication token → 401
2. Invalid token → 401
3. Expired token → 401
4. Valid token for wrong user → 403 (if applicable)
5. Token with insufficient scope → 403
6. Token replay (same token used twice) → Depends on implementation

### Authorization Testing

**Horizontal Access Control:**
- User A should not access User B's resources
- Test by changing resource IDs in requests

**Vertical Access Control:**
- Regular user should not access admin endpoints
- Test by accessing admin endpoints with regular credentials

### Input Validation Testing

**SQL Injection:**
```json
{"username": "admin' OR '1'='1"}
{"username": "admin'; DROP TABLE users; --"}
```

**NoSQL Injection:**
```json
{"username": {"$gt": ""}}
```

**Command Injection:**
```json
{"filename": "file.txt; rm -rf /"}
```

**Path Traversal:**
```
GET /files/../../etc/passwd
```

### Security Headers

| Header | Purpose |
|--------|---------|
| Content-Security-Policy | Prevents XSS by controlling resource loading |
| X-Content-Type-Options | Prevents MIME-type sniffing |
| Strict-Transport-Security | Enforces HTTPS |
| X-Frame-Options | Prevents clickjacking |
| RateLimit-Limit | Informs client of rate limits |

## 5.4 API Performance Testing

### Types of Performance Tests

**Load Testing:**
- Tests API under expected load
- Verifies response times and throughput meet requirements
- **Example:** 100 concurrent users making 10 requests/minute

**Stress Testing:**
- Tests API beyond normal capacity
- Identifies breaking points and graceful degradation
- **Example:** Gradually increase to 1000 concurrent users until errors occur

**Soak Testing:**
- Tests API under sustained load for extended periods
- Identifies memory leaks, connection pool exhaustion
- **Example:** 50 concurrent users for 8 hours

**Spike Testing:**
- Tests sudden, extreme increases in load
- Verifies auto-scaling and recovery
- **Example:** From 10 to 1000 users instantly

### Key Performance Metrics

| Metric | Description | Typical Target |
|--------|-------------|--------------|
| Response time | Time from request to response | p95 < 500ms |
| Throughput | Requests per second | > 1000 RPS |
| Error rate | % of failed requests | < 0.1% |
| Latency | Network delay + processing | p99 < 200ms |
| Concurrent users | Simultaneous active users | As per requirements |

### Performance Testing Tools

| Tool | Type | Best For |
|------|------|----------|
| JMeter | GUI + CLI | Complex scenarios, reporting |
| k6 | CLI (JavaScript) | Developer-friendly, CI/CD |
| Gatling | CLI (DSL) | High throughput, detailed reports |
| Artillery | CLI (JavaScript) | Quick setup, cloud native |
| Locust | Python | Programmable scenarios |

### Example k6 Load Test

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
    stages: [
        { duration: '2m', target: 100 },  // Ramp up
        { duration: '5m', target: 100 },  // Stay at 100
        { duration: '2m', target: 200 },  // Ramp up to 200
        { duration: '5m', target: 200 },  // Stay at 200
        { duration: '2m', target: 0 },    // Ramp down
    ],
    thresholds: {
        http_req_duration: ['p(95)<500'],  // 95% under 500ms
        http_req_failed: ['rate<0.1'],     // Error rate < 0.1%
    },
};

export default function() {
    let res = http.get('https://api.example.com/products');
    check(res, {
        'status is 200': (r) => r.status === 200,
        'response time < 500ms': (r) => r.timings.duration < 500,
    });
    sleep(1);
}
```

## 5.5 Rate Limiting and Throttling Testing

### Rate Limit Types

**Fixed Window:**
- Counter resets at fixed intervals (e.g., every minute)
- Simple but can have burst issues at window boundaries

**Sliding Window:**
- Tracks requests in a rolling time window
- More accurate but more complex

**Token Bucket:**
- Tokens added at fixed rate; each request consumes a token
- Allows bursts up to bucket capacity

### Testing Rate Limits

**Test cases:**
1. Send requests within limit → all succeed (200)
2. Send requests at limit boundary → verify behavior
3. Exceed limit → verify 429 Too Many Requests
4. Verify `Retry-After` header in 429 response
5. Wait for window reset → verify requests succeed again
6. Test rate limit per API key vs. per IP address

## Worked Examples

### Worked Example 1: Authentication Testing

**Scenario:** Test a REST API that uses JWT Bearer tokens.

**Test Cases:**
| Test | Token | Expected |
|------|-------|----------|
| Valid token | Valid, unexpired JWT | 200 OK |
| Missing token | None | 401 Unauthorized |
| Invalid token | "invalid-token" | 401 Unauthorized |
| Expired token | JWT with past `exp` | 401 Unauthorized |
| Tampered token | Valid JWT with modified payload | 401 Unauthorized |
| Wrong algorithm | JWT with `alg: none` | 401 Unauthorized |

### Worked Example 2: Authorization Testing

**Scenario:** An e-commerce API with user and admin roles.

**Horizontal Test:**
```
User A (ID: 100) GET /users/100/orders → 200 (own orders)
User A (ID: 100) GET /users/101/orders → 403 (other user's orders)
```

**Vertical Test:**
```
Regular user GET /admin/users → 403
Admin user GET /admin/users → 200
```

### Worked Example 3: Performance Test Planning

**Scenario:** A product catalog API must handle Black Friday traffic.

**Requirements:**
- Normal load: 100 RPS, p95 < 200ms
- Peak load: 1000 RPS, p95 < 500ms
- No errors above 0.1%

**Test Plan:**

| Test Type | Load | Duration | Purpose |
|-----------|------|----------|---------|
| Load test | 100 RPS | 30 min | Verify normal performance |
| Load test | 1000 RPS | 30 min | Verify peak performance |
| Stress test | Ramp to 5000 RPS | Until failure | Find breaking point |
| Soak test | 500 RPS | 8 hours | Verify stability |
| Spike test | 0 → 2000 RPS instantly | 10 min | Verify auto-scaling |
