# 🪤 Chapter 5 — Tools in Agile: Exam Traps

## 🪤 Trap 1: Thinking Agile Means No Tools

**Trap:** Agile prefers working software over documentation, but tools are still valuable when they support the team. The key is using the right tools, not avoiding tools entirely.

**Example question:**
> Agile teams should avoid all tools because they add bureaucracy. This statement is:
>
> A. True — tools contradict agile values
> B. False — the right tools support agile teams by reducing overhead and increasing visibility
> C. True only for Scrum teams
> D. Required by the Agile Manifesto

**Correct answer:** B — Tools that support collaboration, visibility, and fast feedback are valuable in agile.

---

## 🪤 Trap 2: Selecting Tools Based Only on Popularity

**Trap:** The most popular tool is not always the best fit. Selection should be based on team needs, integration, ease of use, and cost.

**Example question:**
> A team selects a test management tool because it is the most popular in the industry. Six months later, the team finds it too complex and expensive. The mistake was:
>
> A. Using a tool at all
> B. Selecting based on popularity rather than fit for the team's needs
> C. Not buying a more expensive version
> D. Not training the team enough

**Correct answer:** B — Tool selection should match team needs, not just market popularity.

---

## 🪤 Trap 3: Information Radiators Requiring Manual Updates

**Trap:** Information radiators should be self-updating. If someone must manually update them, they will become stale and ignored.

**Example question:**
> A team creates a physical burndown chart that a team member updates every evening. Often, it is outdated by the next morning. The problem is:
>
> A. The chart is not visible enough
> B. Manual updates are unreliable — the chart should pull data automatically from the tool
> C. The team member is not responsible
> D. Burndown charts are not useful

**Correct answer:** B — Information radiators should be self-updating for accuracy.

---

## 🪤 Trap 4: Confusing CI Tools with Test Tools

**Trap:** CI tools (Jenkins, GitHub Actions) orchestrate the pipeline. They are not test management tools. Test management tools (TestRail, Xray) organize and track tests.

**Example question:**
> Jenkins is primarily a:
>
> A. Test management tool for organizing test cases
> B. CI tool for automating builds and test execution
> C. Defect tracking tool
> D. Requirements management tool

**Correct answer:** B — Jenkins is a CI tool that orchestrates builds and test execution.

---

## 🪤 Trap 5: Thinking Tool Selection Is Permanent

**Trap:** Agile teams should be willing to change tools if they no longer fit. Tool selection is not a once-and-done decision.

**Example question:**
> A team's chosen test management tool no longer meets their needs after scaling from 5 to 50 people. They should:
>
> A. Continue using it because changing tools is too disruptive
> B. Evaluate and migrate to a more suitable tool
> C. Stop using any tool and return to spreadsheets
> D. Split into 10 teams to fit the tool

**Correct answer:** B — Agile teams adapt their tools as their needs evolve.
