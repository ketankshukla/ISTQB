# Chapter 2 — Agile Testing: Exam Traps

## Trap 1: Thinking Whole-Team Means Testers Do Everything

**Trap:** Whole-team quality means everyone contributes, not that testers become responsible for all testing tasks. Developers still write unit tests; the team collaborates.

**Example question:**
> In the whole-team approach, the tester:
>
> A. Does all testing because developers are too busy
> B. Collaborates with the team while bringing specialized testing expertise
> C. Is eliminated because the team tests together
> D. Only manages test tools and environments

**Correct answer:** B — Testers collaborate but bring specialized skills. Developers write unit tests; testers focus on higher-level testing.

---

## Trap 2: Confusing Acceptance Criteria with Test Cases

**Trap:** Acceptance criteria define WHAT should be true. Test cases define HOW to verify it. ACs are used to derive test cases but are not the same thing.

**Example question:**
> Acceptance criteria are:
>
> A. The same as test cases
> B. Conditions that a user story must satisfy, from which test cases are derived
> C. Only used by the Product Owner
> D. Written after development is complete

**Correct answer:** B — ACs define conditions for acceptance. Test cases are derived from ACs to verify those conditions.

---

## Trap 3: Thinking "Done" Means Only Development Is Complete

**Trap:** The Definition of Done should include testing, code review, and deployment readiness. "Done" should mean "potentially releasable."

**Example question:**
> A team marks stories as "Done" after developers finish coding. The sprint ends, but many defects are found during testing. The problem is:
>
> A. The testers are too slow
> B. The Definition of Done does not include testing
> C. The sprint is too short
> D. The developers are not skilled

**Correct answer:** B — If "Done" does not include testing, stories are not actually complete. Testing must be part of DoD.

---

## Trap 4: Writing Vague Acceptance Criteria

**Trap:** "Fast," "user-friendly," "good quality" are not testable. Acceptance criteria must be specific and measurable.

**Example question:**
> Which acceptance criterion is testable?
>
> A. "The system should be fast"
> B. "The page loads within 2 seconds"
> C. "The interface should look good"
> D. "Users should find it easy to use"

**Correct answer:** B — "Within 2 seconds" is measurable. "Fast," "look good," and "easy" are subjective.

---

## Trap 5: Ignoring the Definition of Ready

**Trap:** Stories that are not ready before sprint planning lead to blocked work, unclear requirements, and missed testing.

**Example question:**
> A team frequently discovers mid-sprint that stories lack clear requirements or testability. The solution is:
>
> A. Work harder during the sprint
> B. Implement a Definition of Ready to ensure stories are well-defined before sprint planning
> C. Extend the sprint length
> D. Skip testing for unclear stories

**Correct answer:** B — DoR ensures stories are well-defined, have acceptance criteria, and are testable before the sprint starts.
