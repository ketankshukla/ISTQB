# 🪤 Chapter 4 — Quality Metrics and Reporting: Exam Traps

## 🪤 Trap 1: Using Velocity as a Productivity Metric

**Trap:** Velocity is for planning, not comparing teams or measuring productivity. Story points are team-specific.

**Example question:**
> Management compares two teams' velocities to determine which is "more productive." This is:
>
> A. A valid comparison
> B. Invalid — story points are team-specific; velocity is for planning, not comparison
> C. Only wrong if teams use different tools
> D. Only wrong for new teams

**Correct answer:** B — Never compare velocity across teams.

---

## 🪤 Trap 2: Optimizing Metrics Over Outcomes

**Trap:** Surrogation = treating the metric as the goal. Coverage % can increase while real quality decreases.

**Example question:**
> A team increases test coverage from 60% to 95% by writing tests with no assertions. This shows:
>
> A. Excellent quality improvement
> B. Metric gaming — coverage increased but tests don't actually verify behavior
> C. Only faster test execution
> D. Only better documentation

**Correct answer:** B — Metrics must reflect real quality, not just numbers.

---

## 🪤 Trap 3: Reporting Only Good News

**Trap:** Hiding problems prevents improvement. Transparent reporting builds trust and enables action.

**Example question:**
> A quality report shows only green status and positive trends, omitting known issues. The result is:
>
> A. Management confidence
> B. Delayed problem resolution and loss of trust when issues inevitably surface
> C. Only more meetings
> D. Only better bonuses

**Correct answer:** B — Transparent reporting enables early intervention.

---

## 🪤 Trap 4: Using Too Many Metrics

**Trap:** Dashboards with 50 metrics overwhelm and obscure what matters. Focus on a few actionable metrics.

**Example question:**
> A quality dashboard displays 40 different metrics. The team ignores it. The problem is:
>
> A. The team doesn't care about quality
> B. Too many metrics dilute focus; dashboards should highlight a few actionable indicators
> C. Only the color scheme
> D. Only the refresh rate

**Correct answer:** B — Focus on a few meaningful metrics.

---

## 🪤 Trap 5: Confusing Output Metrics with Outcome Metrics

**Trap:** "Tests written" is output; "defects prevented" is outcome. Outcome metrics matter more.

**Example question:**
> A team reports "500 tests written this month" as their quality achievement. A better measure would be:
>
> A. More tests written
> B. Outcome: reduction in defects, faster feedback, or improved coverage of critical paths
> C. Only faster execution
> D. Only better tool usage

**Correct answer:** B — Measure outcomes (quality), not just outputs (activity).
