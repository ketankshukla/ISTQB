# 🪤 Chapter 2 — Acceptance Testing in the SDLC: Exam Traps

## 🪤 Trap 1: Thinking Acceptance Testing Only Happens at the End

**Trap:** In agile and iterative models, acceptance is continuous, not just a final phase.

**Example question:**
> In Scrum, acceptance testing is performed:
>
> A. Only at the end of the project
> B. Continuously throughout each sprint
> C. Only during the first sprint
> D. Never — Scrum doesn't include acceptance testing

**Correct answer:** B — Acceptance criteria are part of each user story and validated during each sprint.

---

## 🪤 Trap 2: Confusing Entry and Exit Criteria

**Trap:** Entry criteria = what must be true to START. Exit criteria = what must be true to STOP/FINISH.

**Example question:**
> "All acceptance tests must be executed and 95% must pass" is:
>
> A. An entry criterion
> B. An exit criterion
> C. A system requirement
> D. A user story

**Correct answer:** B — This defines when acceptance testing is complete = exit criterion.

---

## 🪤 Trap 3: Confusing Sprint Review with Formal Acceptance

**Trap:** Sprint review is an informal demo for feedback. Formal acceptance testing is structured validation against criteria.

**Example question:**
> A stakeholder rejects a user story during sprint review because it doesn't meet acceptance criteria. This is:
>
> A. Only a sprint review finding
> B. An acceptance decision — the story is not accepted
> C. A system test failure
> D. A code review comment

**Correct answer:** B — Rejecting a story for not meeting acceptance criteria IS an acceptance decision, even during sprint review.

---

## 🪤 Trap 4: Ignoring Traceability

**Trap:** Traceability links requirements to tests to defects. Without it, coverage gaps and impact of changes are unknown.

**Example question:**
> A requirements change is proposed. The team cannot identify which tests need updating. The missing practice is:
>
> A. More test automation
> B. Requirement-to-test traceability
> C. More developers
> D. Better project management

**Correct answer:** B — Traceability enables impact analysis when requirements change.

---

## 🪤 Trap 5: Thinking the V-Model Means No Early Testing

**Trap:** The V-model shows corresponding test levels, but test planning and preparation should start early.

**Example question:**
> In the V-model, acceptance test planning should begin:
>
> A. After coding is complete
> B. During requirements specification (the corresponding development phase)
> C. During system testing
> D. After deployment

**Correct answer:** B — Acceptance testing corresponds to requirements, so planning should begin when requirements are defined.
