# CT-AcT Mock Exam 3

**Instructions:** 40 questions, 60 minutes, single best answer.

---

### Q1-Q10

1. Acceptance testing objective: A) Find bugs B) Determine if system satisfies acceptance criteria C) Code review D) Database design — **Determine if system satisfies acceptance criteria**
2. Alpha vs beta: A) Same thing B) Alpha internal, beta external C) Alpha after beta D) Both external — **Alpha internal, beta external**
3. Acceptance criteria should NOT be: A) Testable B) Technical implementation details C) Business-focused D) Specific — **Technical implementation details**
4. OAT tests: A) User interface B) Backup, recovery, operational procedures C) Code quality D) Unit tests — **Backup, recovery, operational procedures**
5. UAT environment should: A) Be development environment B) Mirror production closely C) Have fewer resources D) Be shared with system testing — **Mirror production closely**
6. ATDD tests written: A) After code B) Before development C) During UAT D) After production — **Before development**
7. Three Amigos: A) Three tools B) Business, developer, tester C) Three test types D) Three phases — **Business, developer, tester**
8. Entry criterion example: A) All tests passed B) System testing complete with no critical defects C) Sign-off obtained D) Release notes published — **System testing complete with no critical defects**
9. Exit criterion example: A) Environment ready B) 95% of tests passed C) Users trained D) Requirements defined — **95% of tests passed**
10. Traceability helps: A) Only count tests B) Identify coverage gaps and impact of changes C) Only track developers D) Only measure budget — **Identify coverage gaps and impact of changes**

### Q11-Q20

11. Definition of Done includes: A) Only code written B) Acceptance criteria met and reviewed C) Only unit tests D) Only deployment — **Acceptance criteria met and reviewed**
12. Sprint review vs formal acceptance: A) Same B) Review informal, formal structured validation C) Review longer D) Formal not in agile — **Review informal, formal structured validation**
13. Canary release: A) Full rollout B) Small % users, monitor, expand C) Only test D) Only web — **Small % users, monitor, expand**
14. Pilot deployment purpose: A) Replace testing B) Validate in production with limited group C) Train all at once D) Skip UAT — **Validate in production with limited group**
15. Sign-off with known issues: A) Never allowed B) Allowed with documented risks and plans C) Only for low priority D) Only after 100% automation — **Allowed with documented risks and plans**
16. BDD Given-When-Then: A) Code format B) Business-readable acceptance criteria C) Only for testers D) Only for developers — **Business-readable acceptance criteria**
17. Rule-based criteria best for: A) Complex interactions B) Business rules with clear conditions C) Visual design D) Performance — **Business rules with clear conditions**
18. UAT participants should be: A) Developers B) Business users who know the processes C) Only managers D) Only auditors — **Business users who know the processes**
19. Data profiling in UAT prep: A) Auto-fixes issues B) Understands data structure and quality C) Replaces testing D) Compresses files — **Understands data structure and quality**
20. Automated acceptance tests: A) Replace UAT completely B) Complement but don't replace human UAT C) Are only for agile D) Are not needed — **Complement but don't replace human UAT**

### Q21-Q30

21. CAT vs BAT: A) Same B) CAT contractual, BAT business needs C) CAT cheaper D) BAT not documented — **CAT contractual, BAT business needs**
22. SLA availability 99.9% over 30 days: A) Always passing B) Must be measured continuously C) Only tested once D) Not important — **Must be measured continuously**
23. Outsourced project: Customer should: A) Skip acceptance B) Independently verify deliverables C) Only trust vendor D) Only test at end — **Independently verify deliverables**
24. Liquidated damages: A) Bonus B) Pre-agreed compensation for breach C) Insurance D) Test technique — **Pre-agreed compensation for breach**
25. Service credit for missed SLA: A) Bonus B) Penalty C) Requirement D) Test case — **Penalty**
26. Fixed-price contract acceptance: A) Only at end B) At milestones and completion C) Continuously D) Never — **At milestones and completion**
27. Staff augmentation: A) Vendor manages project B) Contract developers in customer team C) No involvement D) Fixed-price — **Contract developers in customer team**
28. Change control in outsourcing: A) Prevents changes B) Manages scope, avoids disputes C) Increases costs D) Delays project — **Manages scope, avoids disputes**
29. RAT verifies: A) Only function B) Compliance with laws, regulations, standards C) Only performance D) Only UI — **Compliance with laws, regulations, standards**
30. WCAG AA: A) Lowest B) Intermediate commonly required C) Highest D) Not real — **Intermediate commonly required**

### Q31-Q40

31. Keyboard navigation tests: A) Performance B) Accessibility (operable) C) Security D) Load — **Accessibility (operable)**

33. GDPR breach notification: A) 7 days B) 72 hours C) 30 days D) 1 year — **72 hours**
34. PCI-DSS applies to: A) All software B) Payment card data systems C) Only web D) Only mobile — **Payment card data systems**
35. Process compliance: A) Only code B) Processes meet regulatory requirements C) Only coverage D) Only docs — **Processes meet regulatory requirements**
36. External audit by: A) Own employees B) Independent third party C) Only developers D) Only PM — **Independent third party**
37. Surveillance audit: A) Initial certification B) Periodic ongoing compliance C) Only security D) Only financial — **Periodic ongoing compliance**
38. Cucumber: A) Unit framework B) BDD acceptance tool C) Performance tool D) Database tool — **BDD acceptance tool**
39. SpecFlow for: A) Java B) .NET C) Python D) Ruby — **.NET**
40. Service virtualization: A) Replace production B) Simulate dependencies for testing C) Replace mocks D) Eliminate testers — **Simulate dependencies for testing**
