# CT-AcT Exam Strategy

## Exam Format

| Parameter | Value |
|-----------|-------|
| Total questions | 40 |
| Duration | 60 minutes |
| Time per question | 1.5 minutes |
| Pass mark | 65% (26/40) |
| Target score | 32+/40 (80%+) |

---

## Time Budget

| Phase | Time | Action |
|-------|------|--------|
| Quick pass | 45 min | Answer all easy/medium questions |
| Flagged review | 10 min | Return to flagged questions |
| Final check | 5 min | Verify no blanks, review uncertain answers |

**If stuck on a question:** Mark it, move on, return later. Never spend more than 2.5 minutes on one question.

---

## Chapter Weighting and Tactics

| Chapter | Weight | K-Levels | Tactics |
|---------|--------|----------|---------|
| 1 — Introduction | ~15% | K1-K2 | Memorize definitions, understand acceptance vs. other test levels |
| 2 — Acceptance in SDLC | ~15% | K2-K3 | Know how acceptance fits in agile, V-model, incremental |
| 3 — Business Acceptance (BAT) | ~20% | K2-K3 | Master acceptance criteria, UAT, user stories, sign-off |
| 4 — Contract Acceptance (CAT) | ~15% | K2-K3 | Know SLAs, contractual acceptance criteria, supplier relationships |
| 5 — Regulation Acceptance (RAT) | ~15% | K2-K3 | Understand compliance, standards, audit requirements |
| 6 — Tools and Automation | ~20% | K2 | Recognize BDD tools, test management, traceability tools |

---

## K-Level Patterns

**K1 (Remember):** Direct recall of definitions
- Example: "The four types of acceptance testing are..."
- Strategy: Memorize glossary terms precisely

**K2 (Understand):** Comprehend concepts and explain
- Example: "Why is acceptance testing performed by business users rather than developers?"
- Strategy: Focus on "why" and "how" in lessons

**K3 (Apply):** Use knowledge in a scenario
- Example: "Given a set of user stories, write appropriate acceptance criteria."
- Strategy: Practice case studies and worked examples

---

## Common Distractor Patterns

1. **Confusing acceptance criteria with system requirements**
   - Trap: Thinking acceptance criteria are the same as detailed system specs
   - Reality: Acceptance criteria are business-focused, testable statements of what "done" means

2. **Mixing up UAT, BAT, CAT, and RAT**
   - Trap: Confusing the different acceptance testing types
   - Reality: Each has distinct stakeholders, objectives, and criteria

3. **Thinking acceptance testing is only UAT**
   - Trap: Equating acceptance testing with user acceptance testing only
   - Reality: Acceptance testing includes BAT, CAT, RAT, and alpha/beta testing

4. **Confusing entry and exit criteria**
   - Trap: Entry criteria = what must be true to START; exit criteria = what must be true to STOP
   - Reality: Mixing these up is a common exam trap

5. **Thinking acceptance testing happens at the end only**
   - Trap: Believing acceptance testing is a final gate before release
   - Reality: Acceptance activities should be integrated throughout the lifecycle

## Pre-Exam Checklist

- [ ] All 6 chapters completed with practice scores 75%+
- [ ] All 3 case studies completed with model answers reviewed
- [ ] All 3 mock exams completed with scores 28+/40 improving to 32+/40
- [ ] Glossary terms reviewed twice
- [ ] Exam traps from all chapters reviewed
- [ ] Weak areas identified and re-studied

---

## During the Exam

1. **Read the full question** before looking at answers
2. **Eliminate obviously wrong answers** first
3. **Look for key phrases** that distinguish similar concepts
4. **For scenario questions:** Identify the core issue before selecting
5. **Don't overthink** — first instinct is often correct for K1/K2
6. **Mark uncertain questions** and return with fresh perspective

## Post-Exam

- Note any unexpected topics for future learners
- Celebrate — you've prepared thoroughly!
