# 📖 CTAL-TTA Glossary

## ⭐ A

**Acceptance Criteria**
Conditions that a software product must satisfy to be accepted. In non-functional testing, includes measurable performance and security criteria.

## ⭐ B

**Branch Coverage**
A white-box coverage measure where each outcome (true/false) of every decision is executed at least once. Also called decision coverage.

**Buffer Overflow**
A condition where a program writes data beyond the allocated memory buffer, potentially causing crashes or security vulnerabilities.

**Bytecode**
Intermediate code generated by compilers (e.g., Java bytecode, .NET IL) that is executed by a virtual machine. Can be analyzed statically.

## ⭐ C

**Call Graph**
A control flow graph showing which functions call which other functions. Used in static analysis to understand code structure.

**Cyclomatic Complexity**
A measure of the structural complexity of code, calculated as: `V(G) = E - N + 2P` where E = edges, N = nodes, P = connected components. Measures the number of linearly independent paths.

## ⭐ D

**Data Flow Analysis**
Static analysis technique that tracks the flow of data through a program to detect anomalies like uninitialized variables or unused definitions.

**Data Flow Coverage**
Testing that exercises definition-use pairs (DU pairs) in the code. All-uses coverage requires every DU pair to be executed.

**DAST (Dynamic Application Security Testing)**
Security testing of a running application by simulating attacks. Identifies vulnerabilities that are exploitable at runtime.

**Decision Coverage**
See Branch Coverage.

**Def-Use Pair (DU Pair)**
A pair of locations in code where a variable is defined (assigned a value) and then used.

**Dynamic Analysis**
Analysis performed by executing the program and monitoring its behavior (memory usage, performance, resource consumption).

## ⭐ E

**Entry/Exit Criteria**
Entry criteria = conditions to START testing. Exit criteria = conditions to STOP testing.

## ⭐ F

**Fault Injection**
Intentionally introducing errors into a system to test its error-handling and recovery capabilities.

**FMEA (Failure Mode and Effects Analysis)**
A systematic technique for identifying potential failure modes and assessing their impact.

## ⭐ H

**Halstead Metrics**
Software complexity metrics based on the number of operators and operands in code.

## ⭐ I

**IDE (Integrated Development Environment)**
A software application that provides comprehensive facilities for software development, often including static analysis tools.

**Instrumentation**
Adding code to a program to monitor its behavior during execution. Used in dynamic analysis and profiling.

## ⭐ L

**Load Testing**
Testing that evaluates the system's behavior under expected user loads.

**LOC (Lines of Code)**
A basic size metric. Not a direct measure of complexity but often used in productivity and defect density calculations.

## ⭐ M

**MC/DC (Modified Condition/Decision Coverage)**
An advanced coverage criterion where each condition in a decision is shown to independently affect the decision's outcome.

**Memory Leak**
A defect where allocated memory is not properly deallocated, causing the application to consume increasing amounts of memory over time.

**Memory Corruption**
A defect where a program writes to memory locations it should not, potentially causing crashes or security vulnerabilities.

**Mutation Testing**
A technique where the source code is modified (mutated) to check if the test suite detects the introduced defect.

## ⭐ N

**NPath Complexity**
A measure of the number of acyclic execution paths through a function. More precise than cyclomatic complexity for some purposes.

**Non-Functional Testing**
Testing performed to evaluate non-functional aspects: performance, security, reliability, usability, compatibility.

## ⭐ P

**Path Coverage**
A white-box coverage measure where every possible path through the code is executed at least once. Often impractical due to exponential growth of paths.

**Penetration Testing**
Simulating cyberattacks on a system to identify security vulnerabilities.

**Performance Testing**
Testing that evaluates the responsiveness, throughput, and resource utilization of a system under various conditions.

**Profiling**
Dynamic analysis technique that measures execution time, memory usage, and resource consumption of a running program.

## ⭐ R

**Regression Testing**
Testing performed to ensure that changes have not adversely affected existing functionality.

**Risk-Based Testing**
An approach that prioritizes testing based on the assessed risk of failure and the impact of that failure.

## ⭐ S

**SAST (Static Application Security Testing)**
Security testing performed by analyzing the source code without executing it. Identifies potential vulnerabilities in code.

**Scalability Testing**
Testing that evaluates the system's ability to handle increasing workload by adding resources.

**Security Testing**
Testing that evaluates the system's ability to protect data and resources from unauthorized access and attacks.

**Source Code**
Human-readable code written in a programming language. The input to static analysis tools.

**Statement Coverage**
A white-box coverage measure where every executable statement in the code is executed at least once.

**Static Analysis**
Analysis of code without executing it. Finds defects, measures complexity, enforces standards, and identifies security vulnerabilities.

**Stress Testing**
Testing that evaluates the system's behavior under extreme load conditions, often beyond expected capacity.

## ⭐ T

**Technical Risk**
A risk related to the technical aspects of the software: complexity, technology, architecture, performance, security.

**Test Coverage**
The degree to which specified coverage items have been exercised. Can apply to requirements, code, risks, or design elements.

**Testability**
The degree to which a system or component facilitates testing. Influenced by code structure, observability, and controllability.

**Token**
In static analysis, a single element of source code (keyword, operator, identifier) used in metrics like Halstead's.

## ⭐ U

**Unit Testing**
Testing of individual software components in isolation, typically by the developer who wrote the code.

**Usability Testing**
Testing that evaluates how easy it is for users to learn and use the system.

## ⭐ V

**Verification**
Confirmation by examination that specified requirements have been fulfilled. Static techniques (reviews, static analysis) are verification activities.

**Vulnerability**
A weakness in a system that can be exploited to cause harm. Security testing aims to identify vulnerabilities.

## ⭐ W

**White-Box Testing**
Testing based on the internal structure of the code. Includes control flow testing, data flow testing, and coverage analysis.
