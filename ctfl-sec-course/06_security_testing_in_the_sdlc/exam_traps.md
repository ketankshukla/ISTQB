# 🪤 Chapter 6 — Security Testing in the SDLC: Exam Traps

## 🪤 Trap 1: Security Testing is a Separate Phase

**Trap:** Security must be integrated throughout the SDLC, not a single phase.

**Example question:**
> Security testing is performed as a dedicated phase before release. This is:
>
> A. Correct — it's the final quality gate
> B. Incorrect — security should be integrated throughout all phases (SSDLC), not just one phase at the end
> C. Only for waterfall
> D. Only for regulated industries

**Correct answer:** B — Integrated throughout, not a phase.

---

## 🪤 Trap 2: Shift-Left Means Only Moving Testing Earlier

**Trap:** Shift-left also includes security requirements, design, and education.

**Example question:**
> Shift-left security only means running security tests earlier in development. This is:
>
> A. Correct — testing earlier is the goal
> B. Incorrect — shift-left includes requirements, design, threat modeling, and developer education, not just testing
> C. Only for agile
> D. Only for DevOps

**Correct answer:** B — Shift-left is broader than testing.

---

## 🪤 Trap 3: DevSecOps Means Security Team Manages Everything

**Trap:** DevSecOps means shared responsibility across dev, ops, and security.

**Example question:**
> In DevSecOps, the security team is solely responsible for all security activities. This is:
>
> A. Correct — security is their job
> B. Incorrect — DevSecOps means shared responsibility: developers write secure code, ops secure infrastructure, security provides guidance and tools
> C. Only for enterprises
> D. Only for startups

**Correct answer:** B — Shared responsibility.

---

## 🪤 Trap 4: Automated Security Checks Replace Manual Review

**Trap:** Automation complements but does not replace human judgment.

**Example question:**
> In a DevSecOps pipeline, automated security checks eliminate the need for manual security review. This is:
>
> A. Correct — automation handles everything
> B. Incorrect — automation catches known issues; manual review is still needed for architecture, threat modeling, and complex logic
> C. Only for mature pipelines
> D. Only for new pipelines

**Correct answer:** B — Automation + manual together.

---

## 🪤 Trap 5: Security Stories Slow Down Agile

**Trap:** Security stories are part of normal velocity, not overhead.

**Example question:**
> Adding security user stories to sprints slows down development and should be avoided. This is:
>
> A. Correct — security is overhead
> B. Incorrect — security stories are essential product features; addressing them early prevents costly fixes later
> C. Only for short sprints
> D. Only for long sprints

**Correct answer:** B — Security stories are essential.
