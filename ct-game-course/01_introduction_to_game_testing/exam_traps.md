# Chapter 1 — Introduction to Game Testing: Exam Traps

## Trap 1: Game Testing is Just Playing Games

**Trap:** Game testing is structured, documented, and repeatable — not random gameplay.

**Example question:**
> Game testing is primarily about playing the game to find fun elements. This is:
>
> A. Correct — fun is the main goal
> B. Incorrect — game testing is structured verification of requirements, functionality, and quality; playtesting evaluates fun
> C. Only true for indie games
> D. Only true for AAA games

**Correct answer:** B — Testing ≠ playing for fun.

---

## Trap 2: All Game Genres Test the Same Way

**Trap:** Different genres have different testing priorities (RPG = save/load, FPS = hit detection).

**Example question:**
> A tester applies FPS testing methods (hit detection, physics) to a puzzle game. The likely result is:
>
> A. Perfect coverage
> B. Missing critical puzzle-specific tests (solution validation, hint system, edge cases)
> C. Only slower testing
> D. Only more expensive testing

**Correct answer:** B — Match testing to genre.

---

## Trap 3: Compliance Testing is Optional

**Trap:** Platform holders require compliance testing; games cannot release without certification.

**Example question:**
> A team skips compliance testing to save time before launch. The consequence is:
>
> A. Faster launch
> B. Platform holder will reject submission; game cannot be released on that platform
> C. Only a fine
> D. Only bad reviews

**Correct answer:** B — Compliance is mandatory for release.

---

## Trap 4: Consoles and PC Have the Same Testing Requirements

**Trap:** Consoles require certification; PC has more hardware variability.

**Example question:**
> Console and PC testing require exactly the same approach. This is:
>
> A. Correct — testing is testing
> B. Incorrect — console requires certification (TRC/Lotcheck); PC requires broad hardware compatibility testing
> C. Only true for AAA
> D. Only true for indie

**Correct answer:** B — Platform differences drive testing differences.

---

## Trap 5: Playtesting Replaces QA Testing

**Trap:** Playtesting evaluates fun and UX; QA testing finds bugs. Both are needed.

**Example question:**
> A team does extensive playtesting and concludes no QA testing is needed. The risk is:
>
> A. No risk — players will find all issues
> B. Systematic bugs, edge cases, and compliance issues may be missed; playtesting and QA serve different purposes
> C. Only that testing is slower
> D. Only that testing is more expensive

**Correct answer:** B — Playtesting and QA are complementary.
