# Chapter 6 — Test Automation and Tools: Exam Traps

## Trap 1: Automating Everything

**Trap:** Games need extensive manual/exploratory testing for feel, fun, and emergent behavior.

**Example question:**
> A game project aims to automate 100% of testing. The risk is:
>
> A. No risk — automation is always better
> B. Gameplay feel, exploratory discovery, and UX evaluation cannot be automated and will be missed
> C. Only that it costs more
> D. Only that it takes longer

**Correct answer:** B — Automation complements but doesn't replace manual testing.

---

## Trap 2: Screenshot Comparison with Zero Tolerance

**Trap:** Particle effects, animations, and timing differences cause false positives.

**Example question:**
> Screenshot comparison flags every build due to particle effect differences. The solution is:
>
> A. Disable screenshot comparison
> B. Use tolerance levels and mask dynamic regions; not all differences are regressions
> C. Only compare UI elements
> D. Only compare on one GPU

**Correct answer:** B — Tolerance and masking handle acceptable variance.

---

## Trap 3: CI/CD is Only for Code

**Trap:** Game CI/CD must also build assets, run smoke tests, and validate content.

**Example question:**
> A game CI pipeline only compiles code. Missing checks likely include:
>
> A. Nothing — code compilation is sufficient
> B. Asset validation, smoke tests, and build packaging for target platforms
> C. Only that it's slower
> D. Only that it's cheaper

**Correct answer:** B — Game CI = code + assets + tests + packaging.

---

## Trap 4: Telemetry Doesn't Need Testing

**Trap:** Incorrect telemetry leads to bad business decisions; events must be verified.

**Example question:**
> Analytics events are automatically generated by the engine and don't need testing. This is:
>
> A. Correct — engines handle analytics
> B. Incorrect — event triggers, parameters, and data accuracy must be verified to ensure correct business decisions
> C. Only for small studios
> D. Only for offline games

**Correct answer:** B — Telemetry needs validation.

---

## Trap 5: Bug Severity is the Only Priority Factor

**Trap:** Frequency, impact, and fix cost also determine priority.

**Example question:**
> A rare crash (1 in 1000 players) is automatically low priority. The better approach is:
>
> A. Correct — rarity means low impact
> B. Consider frequency, player impact, and reproduction; a rare crash at a critical moment (e.g., final boss) may be high priority
> C. Only for AAA games
> D. Only for multiplayer

**Correct answer:** B — Consider context, not just frequency.
