# Chapter 1 — Testing Process: Exam Traps

## Trap 1: Confusing Test Monitoring with Test Control

**Trap:** Monitoring collects information. Control takes action based on that information.

**Example question:**
> A test manager reviews test execution reports and notices that 30% of test cases are failing. The manager schedules a meeting with the development team to discuss defect resolution. This is an example of:
>
> A. Test monitoring
> B. Test control
> C. Test planning
> D. Test analysis

**Correct answer:** B — Taking corrective action (scheduling a meeting to address failures) is test control. Merely reviewing reports would be monitoring.

---

## Trap 2: Confusing Entry Criteria with Exit Criteria

**Trap:** Entry criteria must be met to START testing. Exit criteria must be met to STOP testing.

**Example question:**
> "All high-priority test cases have been executed and passed" is an example of:
>
> A. An entry criterion for system testing
> B. An exit criterion for system testing
> C. A suspension criterion
> D. A test objective

**Correct answer:** B — This describes when testing can be considered complete, so it is an exit criterion.

---

## Trap 3: Confusing Test Policy, Strategy, Plan, and Approach

**Trap:** Policy = organizational. Strategy = program/project. Plan = detailed. Approach = implementation.

**Example question:**
> A document stating that "all projects shall perform risk-based testing with at least 80% requirements coverage" is a:
>
> A. Test plan
> B. Test strategy
> C. Test policy
> D. Test procedure

**Correct answer:** C — This is an organizational-level directive applicable to all projects, making it a test policy.

---

## Trap 4: Thinking Test Completion Is Just Reporting Results

**Trap:** Test completion includes evaluating objectives, archiving testware, and documenting lessons learned — not just reporting.

**Example question:**
> Test completion activities include all of the following EXCEPT:
>
> A. Creating a test summary report
> B. Evaluating whether test objectives were met
> C. Writing new test cases for the next release
> D. Archiving testware for future use

**Correct answer:** C — Writing new test cases for the next release is future planning, not test completion. Test completion focuses on closing the current test effort.

---

## Trap 5: Confusing Product Risks with Project Risks

**Trap:** Product risks relate to the quality of the software. Project risks relate to the success of the project.

**Example question:**
> "The system may fail under peak load conditions" is:
>
> A. A project risk
> B. A product risk
> C. A test risk
> D. A business risk

**Correct answer:** B — This is a risk about the quality of the product (performance under load), making it a product risk.
