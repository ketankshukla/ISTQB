---

## Answers and Rationales

### Q1
**Answer: C**

**Rationale:** Test policy is organizational-level. Test strategy is program/project-level. Test plan is detailed project-level. Test procedure is step-by-step execution instructions.

### Q2
**Answer: B**

**Rationale:** Test monitoring tracks progress. Test control takes corrective action. Test planning defines objectives. Test completion closes the test effort.

### Q3
**Answer: B**

**Rationale:** Allocating additional resources to address a problem is a control action. Monitoring would be simply noticing the slow resolution.

### Q4
**Answer: B**

**Rationale:** Environment availability is a condition that must be met BEFORE testing begins — an entry criterion.

### Q5
**Answer: B**

**Rationale:** Test planning is the phase where the test plan is developed. It guides subsequent phases.

### Q6
**Answer: B**

**Rationale:** Adjusting scope to address schedule pressure is a control action. Monitoring would just be observing the schedule slippage.

### Q7
**Answer: B**

**Rationale:** System crashing under load is a quality issue with the product — a product risk. The others are project risks (staffing, environment, budget).

### Q8
**Answer: B**

**Rationale:** Test design transforms conditions into cases. Test analysis identifies conditions. Test implementation prepares procedures. Test execution runs the tests.

### Q9
**Answer: B**

**Rationale:** Archiving testware is a test completion activity. Writing cases is design/implementation. Executing is execution.

### Q10
**Answer: B**

**Rationale:** Test strategy describes test levels for a program. Test plan is more detailed. Test policy is organizational. Test approach is the implementation.

### Q11
**Answer: B**

**Rationale:** Low requirements coverage (60%) indicates that not all requirements have test cases — more test design is needed. Execution progress, defects found, and staffing don't indicate design gaps.

### Q12
**Answer: B**

**Rationale:** Policy (organizational) → Strategy (program/project) → Plan (detailed). This is the hierarchical relationship.

### Q13
**Answer: B**

**Rationale:** Resolved critical defects indicate testing can be considered complete — an exit criterion.

### Q14
**Answer: C**

**Rationale:** Test implementation includes preparing data and environment. Analysis identifies conditions. Design creates cases. Execution runs tests.

### Q15
**Answer: B**

**Rationale:** Decreasing defect find rate with increasing execution may indicate tests are not effective, the application is stabilizing, or tests are too simple. Investigation is needed before action.

### Q16
**Answer: B**

**Rationale:** Scope, approach, resources, and schedule are standard test plan contents. Code review checklists, stand-up notes, and marketing strategy are not test plan items.

### Q17
**Answer: B**

**Rationale:** Spending 60% of budget for 40% completion at 50% schedule is a negative variance. Investigation and control actions are warranted.

### Q18
**Answer: B**

**Rationale:** Project risk = risk to project objectives (schedule, budget, scope). Product risk = risk to product quality.

### Q19
**Answer: C**

**Rationale:** Test implementation creates procedures, assembles suites, prepares data and environment.

### Q20
**Answer: B**

**Rationale:** Test environment availability is an entry criterion (needed to START), not an exit criterion (needed to STOP).
