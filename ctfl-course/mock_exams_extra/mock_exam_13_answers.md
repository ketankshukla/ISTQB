# 📝 Mock Exam 13 — Answer Key (Extended Practice Set)

Each answer shows the **correct option**, a **rationale**, the **Learning Objective (LO)** it tests, and the **K-level**.

---

### ✔️ Q1 — Answer: A  *(FL-1.1.1, K1)*
A core objective of testing is to **reduce risk to an acceptable level** — not to maximize risk, avoid documentation, or replace developers.

### ✔️ Q2 — Answer: B  *(FL-1.2.2, K1)*
**Quality management** includes both **quality assurance** (process-oriented) and **quality control** (product-oriented, including testing).

### ✔️ Q3 — Answer: B  *(FL-1.3.1, K2)*
The **pesticide paradox**: repeating the same tests eventually stops finding new defects, so tests must be reviewed and updated.

### ✔️ Q4 — Answer: B  *(FL-1.2.3, K2)*
**Root cause analysis** finds the **underlying reason** for defects so similar future defects can be prevented — going beyond the visible symptom.

### ✔️ Q5 — Answer: B  *(FL-1.4.3, K2)*
The **test completion report** is produced during the **test completion** activity, at the end of a test effort.

### ✔️ Q6 — Answer: A  *(FL-1.4.1, K2)*
Running tests and recording outcomes are part of **test execution**.

### ✔️ Q7 — Answer: B  *(FL-1.4.5, K2)*
Deciding the **overall approach and resourcing** is a **test management** responsibility.

### ✔️ Q8 — Answer: B  *(FL-1.4.2, K2)*
**Context-dependent testing** means a **pacemaker** (safety-critical) is tested far more rigorously than a casual game — rigor matches risk and context.

### ✔️ Q9 — Answer: A  *(FL-2.1.3, K1)*
In **test-first** approaches, **tests/examples guide the implementation** — they are written before or alongside the code.

### ✔️ Q10 — Answer: B  *(FL-2.1.2, K2)*
A good practice across SDLCs is to **detect defects as early as possible**, reducing the cost of fixing them.

### ✔️ Q11 — Answer: B  *(FL-2.1.1, K2)*
In **iterative** development, testing happens **within each iteration**, with the **regression scope growing** as more functionality is delivered.

### ✔️ Q12 — Answer: B  *(FL-2.2.3, K2)*
**Regression testing** ensures a **new feature has not broken existing functionality** — checking for unintended side effects.

### ✔️ Q13 — Answer: B  *(FL-2.2.2, K2)*
**Accessibility** is a quality characteristic, so accessibility testing is **non-functional testing**.

### ✔️ Q14 — Answer: A  *(FL-2.2.1, K2)*
**Component integration testing** is typically performed **after component testing and before system testing**, focusing on interfaces between integrated components.

### ✔️ Q15 — Answer: B  *(FL-3.2.3, K1)*
The **author** creates the work product and **fixes the agreed defects**. Leading, recording, and budget approval belong to other roles.

### ✔️ Q16 — Answer: B  *(FL-3.1.1, K1)*
**Static analysis** finds **coding-standard violations and certain defects without executing** the code. Performance, satisfaction, and latency need dynamic measurement.

### ✔️ Q17 — Answer: B  *(FL-3.1.3, K2)*
**Static** testing examines work products **without execution**; **dynamic** testing **executes** the software. (Option A reverses them.)

### ✔️ Q18 — Answer: B  *(FL-3.2.2, K2)*
The **communication and analysis** step produces a **classified list of issues/defects** and decisions on which to address.

### ✔️ Q19 — Answer: B  *(FL-4.1.1, K2)*
Ensuring every statement executes at least once is a **white-box** objective — **statement coverage**.

### ✔️ Q20 — Answer: B  *(FL-4.3.3, K2)*
100% statement coverage can **miss untested decision outcomes** (e.g., the false branch of an IF with no ELSE), so **branch coverage** adds assurance.

### ✔️ Q21 — Answer: B  *(FL-4.4.1, K2)*
**Error guessing** is best used to **complement systematic techniques**, adding tests for likely problem areas they might miss.

### ✔️ Q22 — Answer: B  *(FL-4.4.2, K2)*
**Exploratory testing** depends heavily on the **tester's skill, knowledge, and creativity**, rather than pre-scripted steps.

### ✔️ Q23 — Answer: A  *(FL-4.5.1, K2)*
The **Three Amigos** are the **business, development, and testing** perspectives that collaborate on user stories.

### ✔️ Q24 — Answer: B  *(FL-4.5.2, K2)*
Poorly written acceptance criteria create **ambiguity**, risking building or testing the **wrong thing**.

### ✔️ Q25 — Answer: C  *(FL-4.2.1, K3)*
Minimum partitions: below 1 (invalid), 1–5 (valid), above 5 (invalid) = **3**.

### ✔️ Q26 — Answer: A  *(FL-4.2.2, K3)*
**Three-value BVA** at the upper boundary 5 tests the boundary and both neighbors: **4, 5, 6**.

### ✔️ Q27 — Answer: B  *(FL-4.2.3, K3)*
When conditions are **not independent**, some combinations are infeasible or "don't care," so the simplified table has **fewer than the full 4 rules**.

### ✔️ Q28 — Answer: B  *(FL-4.2.4, K3)*
The number of test cases for **0-switch coverage** is **as many paths as needed to traverse all valid transitions at least once** — not fixed to the number of states.

### ✔️ Q29 — Answer: B  *(FL-4.3.1, K3)*
Straight-line code with **no branches** is fully covered by a **single** test case, regardless of the number of statements.

### ✔️ Q30 — Answer: A  *(FL-5.3.1, K1)*
"Defects found per test level" is a **test metric** used to monitor and report on testing.

### ✔️ Q31 — Answer: B  *(FL-5.1.1, K2)*
A **master test plan** provides an **overall plan spanning multiple levels/activities**, whereas a level test plan covers a single level.

### ✔️ Q32 — Answer: B  *(FL-5.1.3, K2)*
"No critical defects open and 95% passed" describes **exit criteria** — conditions for stopping testing.

### ✔️ Q33 — Answer: B  *(FL-5.2.3, K2)*
**Product risk analysis** is done **early and revisited throughout**, continuously guiding test prioritization.

### ✔️ Q34 — Answer: B  *(FL-5.2.2, K2)*
Losing a **key domain expert** is a **project risk** — it threatens the project's ability to deliver, rather than being a product flaw.

### ✔️ Q35 — Answer: A  *(FL-5.4.1, K2)*
**Configuration management** ensures the **exact versions** of software and testware are known and **reproducible**.

### ✔️ Q36 — Answer: A  *(FL-5.1.5, K3)*
Highest first: **R3 (10)**, then **R4 (7)**, then the two **4s** in original order (**R1, R2**) → R3, R4, R1, R2.

### ✔️ Q37 — Answer: B  *(FL-5.2.1, K3)*
Risk level = likelihood × impact = 2 × 2 = **4**.

### ✔️ Q38 — Answer: A  *(FL-5.5.1, K3)*
**Priority** indicates the **urgency** and therefore the **order** in which defects should be fixed (severity indicates impact).

### ✔️ Q39 — Answer: A  *(FL-6.1.1, K1)*
A tool that **simulates many concurrent users** is a **performance/load testing tool**.

### ✔️ Q40 — Answer: B  *(FL-6.2.1, K2)*
