# 📝 Mock Exam 10 — Answer Key (Extended Practice Set)

Each answer shows the **correct option**, a **rationale**, the **Learning Objective (LO)** it tests, and the **K-level**.

---

### ✔️ Q1 — Answer: D  *(FL-1.1.1, K1)*
Testing evaluates work products, finds defects, and reduces risk — but it **cannot guarantee a perfect product** (that would contradict the principle that testing shows the presence, not the absence, of defects).

### ✔️ Q2 — Answer: B  *(FL-1.5.2, K1)*
The **whole-team approach** means the **whole team collaborates on and shares responsibility for quality**, rather than delegating it to testers or outsourcing it.

### ✔️ Q3 — Answer: B  *(FL-1.3.1, K2)*
This principle states that **passing tests cannot prove the absence of defects** — testing can only reveal defects that are present.

### ✔️ Q4 — Answer: B  *(FL-1.2.3, K2)*
An **error (mistake)** is the human action that introduces a **defect**. The failure is the observable result; root-cause analysis is an activity, not the mistake itself.

### ✔️ Q5 — Answer: B  *(FL-1.4.1, K2)*
**Test design** creates the test cases and test procedures from the test conditions identified during analysis.

### ✔️ Q6 — Answer: B  *(FL-1.4.4, K2)*
**Bidirectional traceability** links **test basis items to test conditions/cases and back to results**, enabling coverage analysis and impact assessment in both directions.

### ✔️ Q7 — Answer: B  *(FL-1.4.5, K2)*
A principal task of the **testing role** is **implementing and executing** tests (along with analysis and design). Budgets, strategy, and staffing belong to management.

### ✔️ Q8 — Answer: A  *(FL-1.2.1, K2)*
Testing is necessary to help meet **contractual, legal, or regulatory requirements** and to **reduce operational risk** — not to replace development or guarantee perfection.

### ✔️ Q9 — Answer: B  *(FL-2.1.3, K1)*
**TDD** follows the **red-green-refactor** cycle: write a failing test, write just enough code to pass it, then refactor.

### ✔️ Q10 — Answer: B  *(FL-2.1.2, K2)*
**Reviewing the test basis early** to find defects is a good practice in **any** SDLC. The other options are needless restrictions.

### ✔️ Q11 — Answer: B  *(FL-2.1.4, K2)*
In **DevOps**, **continuous testing** runs **automated tests continuously** as part of integration and delivery, giving rapid feedback on every change.

### ✔️ Q12 — Answer: B  *(FL-2.2.1, K2)*
In the **V-model**, each development level has a **corresponding test level** (e.g., system design ↔ integration testing), with tests planned early and executed later.

### ✔️ Q13 — Answer: B  *(FL-2.2.2, K2)*
Checking encryption at rest is **security testing**, a **non-functional** test type concerned with a quality characteristic.

### ✔️ Q14 — Answer: D  *(FL-2.3.1, K2)*
The **first release of a brand-new product** is initial development, **not** maintenance. Maintenance testing is triggered by modification, migration, or retirement.

### ✔️ Q15 — Answer: B  *(FL-3.2.3, K1)*
The **reviewer(s)** examine the work product to **identify defects**. The author owns/fixes it, the scribe records, and the manager has a managerial role.

### ✔️ Q16 — Answer: B  *(FL-3.1.1, K1)*
A key advantage of **static testing** is that it can be applied **early — even before any code exists** (e.g., reviewing requirements).

### ✔️ Q17 — Answer: B  *(FL-3.2.2, K2)*
The **communication and analysis** activity (which may include a **review meeting**) is where findings are discussed and consolidated.

### ✔️ Q18 — Answer: B  *(FL-3.2.4, K2)*
A **walkthrough** is led by the **author** to **share knowledge, build understanding, and find defects**. Formal exit criteria and metrics are characteristic of inspections.

### ✔️ Q19 — Answer: B  *(FL-4.1.1, K2)*
**Exploratory testing** is an **experience-based** technique. BVA and decision tables are black-box; branch testing is white-box.

### ✔️ Q20 — Answer: B  *(FL-4.3.3, K2)*
**White-box coverage tools** show **which code has not been exercised**, revealing gaps in structural coverage.

### ✔️ Q21 — Answer: B  *(FL-4.4.1, K2)*
Testing 29 February specifically, based on experience that date logic often fails there, is **error guessing**.

### ✔️ Q22 — Answer: B  *(FL-4.4.2, K2)*
Exploratory testing is commonly structured with **session-based test management**, adding charters, time-boxing, and debriefs/reporting.

### ✔️ Q23 — Answer: B  *(FL-4.5.2, K2)*
Good acceptance criteria are **testable** and **clearly define "done"**, so the team agrees on completion and can derive acceptance tests.

### ✔️ Q24 — Answer: B  *(FL-4.5.1, K2)*
A user story is a brief **placeholder for a conversation**; the details emerge through discussion (the second of the 3 C's).

### ✔️ Q25 — Answer: C  *(FL-4.2.1, K3)*
There are **3 valid partitions**, one per weight tier: 0–1 kg, >1–5 kg, and >5 kg.

### ✔️ Q26 — Answer: A  *(FL-4.2.2, K3)*
**Three-value BVA** at the lower boundary 100 tests the boundary and both neighbors: **99, 100, 101**.

### ✔️ Q27 — Answer: B  *(FL-4.2.3, K3)*
Two binary conditions give 2² = **4 rules**; the decision table ensures **all combinations** of the conditions are considered, not just the happy path.

### ✔️ Q28 — Answer: B  *(FL-4.2.4, K3)*
There are **4 valid transitions** (insert coin, cancel, dispense, deliver); 0-switch coverage requires exercising all **4**.

### ✔️ Q29 — Answer: B  *(FL-4.3.2, K3)*
Each decision needs a TRUE and a FALSE outcome. Since the two decisions are independent, **2** test cases suffice — one setting both decisions TRUE, the other setting both FALSE — covering all four branch outcomes.

### ✔️ Q30 — Answer: B  *(FL-5.1.6, K1)*
The **test pyramid** favors **many unit tests** at the base and **fewer UI tests** at the top, optimizing speed, cost, and stability.

### ✔️ Q31 — Answer: A  *(FL-5.1.1, K2)*
A **test plan** defines **how testing supports the project's objectives** and how it will be organized — it is not a code store, salary list, or replacement for requirements.

### ✔️ Q32 — Answer: B  *(FL-5.1.3, K2)*
A deployed build with passing smoke tests is a typical **entry criterion** that must be met before deeper testing begins.

### ✔️ Q33 — Answer: B  *(FL-5.2.1, K2)*
**Risk level** is determined by combining the **likelihood** of occurrence with the **impact** if it occurs.

### ✔️ Q34 — Answer: B  *(FL-5.2.2, K2)*
A shortage of **skilled testers** is a **project risk** — it threatens the project's ability to deliver, rather than being a flaw in the product.

### ✔️ Q35 — Answer: B  *(FL-5.3.2, K2)*
A **progress (interim) report** keeps stakeholders informed of **status, progress, and risks during testing**, unlike the end-of-activity completion report.

### ✔️ Q36 — Answer: A  *(FL-5.1.5, K3)*
Highest first: **R4 (10)**, then the two **7s** (R2 before R3 by the tie-break), then **R1 (3)** → **R4, R2, R3, R1**.

### ✔️ Q37 — Answer: B  *(FL-5.2.1, K3)*
Risk level = likelihood × impact = 5 × 5 = **25** (the maximum on this scale).

### ✔️ Q38 — Answer: B  *(FL-5.5.1, K3)*
The **expected vs actual result** fields make the **discrepancy** (the essence of the defect) explicit, so others can judge whether and how the behavior is wrong.

### ✔️ Q39 — Answer: A  *(FL-6.1.1, K1)*
**Performance/load testing tools** are a recognized category of test tool. The other options are not software test tools.

### ✔️ Q40 — Answer: B  *(FL-6.2.1, K2)*
Automation is **most beneficial for stable, repetitive tests** and **requires ongoing maintenance**; it does not suit everything, guarantee defect-free releases, or remove the need for a strategy.
